Conditional Requests¶

Overview¶

This module provides support for conditional request handling, based on ETags, as defined in RFC 7232.

There are two use cases for conditional requests:

  1. For safe methods (ie. OPTIONS, GET, and HEAD), conditional requests can be used by caching clients — including caching forward proxies — to download the resource representation only if it has changed since the cached version. In this case, the client adds an If-None-Match: header and includes the ETag(s) of the cached representation(s).
  2. For unsafe methods (ie. DELETE, PATCH, POST, and PUT), conditional requests can be used to avoid lost updates. For this use case, there are two scenarios:
    1. When performing an unsafe request, the client adds an If-Match: header with the ETag of the resource as last seen by the client. This guarantees that the client doesn’t accidentally delete or overwrite a newer version of the resource.
    2. When performing a PUT request with the intention of creating a new resource, the client adds an If-None-Match: * header to assert that no resource yet exists at the given request URI.

Note

Even if ETags is prohibitively expensive for your resources, this module can still handle conditional requests based on the presence or absence of your (dynamic) resource. In this case, only use case 2.b. is really supported.

ETagMixin¶

If you want conditional request handling for one of your views, use ETagMixin, and implement abstract method etag. The mixin provides a method assert_preconditions, which gets called automatically by the framework for GET and HEAD requests. For other requests, you need to call this method yourself at some appropriate moment during request handling.

Helpers for ETag creation¶

To facilitate the creation of valid ETags for your resources, this library provides the following helpers:

  • etaggify() converts a string to a syntactically valid ETag.
  • etag_from_int() and etag_from_float() convert integers and floating point values to very compact ETags.
  • ETagGenerator can create a unique ETag from any (complex) value, using canonical JSON encoding and SHA3_224 hashing.

_assert_if_none_match(request, etag, require, allow_weak)[source]¶

The If-None-Match: header is used in two scenarios:

  1. GET requests by a caching client. In this case, the client will normally provide a list of (cached) ETags.
  2. PUT requests, where the client intends to create a new resource and wants to avoid overwriting an existing resource. In this case, the client will normally provide only the asterisk “*” character.
Raises:
  • web.HTTPBadRequest – If the request header is syntactically incorrect.
  • web.HTTPPreconditionRequired – If parameter require is True and no precondition was provided by the client.
  • web.HTTPPreconditionFailed – If the precondition failed and the request is unsafe.
  • web.HTTPNotModified – If the precondition failed and the request is safe.
